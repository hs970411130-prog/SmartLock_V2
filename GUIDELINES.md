# GUIDELINES.md — SmartLock_v2

编码行为准则，降低 LLM 辅助编码中的常见失误。与项目结构约束 `AGENTS.md` 互补。

**取舍声明：** 本文档偏向谨慎而非速度。琐碎任务可灵活判断。

## 1. 先想清楚再写代码

**不要假设。不要隐藏困惑。主动暴露取舍。**

动手之前：
- 明确写出你的假设。如果不确定，提问
- 如果存在多种解读，列出来——不要悄悄选一种
- 如果有更简单的方案，说出来。有理有据地反驳需求
- 如果有东西不清楚，停下来。指出哪里困惑。问清楚

### 歧义判定标准（以下任一模糊即属歧义，必须前置确认）

- 参数取值范围不清（如"超时时间"未给具体毫秒数）
- 时序约束不明确（如"A 完成后 B 触发"未说明 B 的截止时间）
- 异常处理路径未定义（如"指纹识别失败后重试几次"）
- 存储容量边界模糊（如"最多存多少条"未给定上限）
- 外设故障容错策略未说明（如"SPI 掉线后是重试还是报错"）

### 紧急阻断 bug 例外（满足任意一条即可直接修复，无需确认）

1. 整机完全无法开锁（继电器不动作）
2. 上电反复硬件看门狗 / 独立看门狗复位
3. 核心外设（指纹/RFID/继电器）上电彻底失联，无任何交互
4. MQTT/WiFi 崩溃导致整机卡死调度

以下**不属于**紧急阻断，仍需走确认流程：纯界面展示 bug、日志排版问题、次要提示音异常。修复后补记录即可。

## 2. 极简优先

**解决问题的最少代码。不写预判性功能。**

- 不写需求之外的功能
- 不为单次使用创建抽象层
- 不主动加「灵活性」「可配置性」
- 不走预测式编程（"将来可能用到"）
- 如果写了 200 行能缩减到 50，重写

自问：「资深嵌入式工程师会觉得这段代码过度复杂吗？」如果是，简化。

**嵌入式例外（真实风险，不可省略）：**
- 硬件层（drv_*.c）：保留必要的超时返回、错误码——SPI 超时、I2C NACK、Flash 写入失败
- 应用层公开接口：被中断回调/队列接收后调用的函数，第一个参数必须判空
- 内部私有函数：不需要防御性判空

## 3. 手术式改动

**只碰必须改的。只清理自己造成的烂摊子。**

修改已有代码时：
- 不要「顺手优化」相邻代码、注释、格式
- 不要重构没坏的东西
- 匹配现有代码风格，哪怕你觉得自己的写法更好
- **禁止顺手统一格式化**（全文件缩进、换行风格修改），避免无意义 diff 干扰评审
- 如果发现无关的死代码，提一句——不要主动删除

仅清理自己造成的遗留：
- 删除你的改动导致不再使用的 include / 变量 / 函数
- 不删除改动前就存在的死代码

**改动超过 3 个文件时，拆分任务。** 例外：修改顶层公共配置/类型头文件（app_config.h、app_types.h、app_ipc.h）导致的多文件联动，不计入限制，但 commit message 必须标注"公共头文件变更联动"。提交前用 `git diff --stat` 检查。

检验标准：每一行改动都能直接追溯到需求。

## 4. 目标驱动执行

**定义成功标准。循环验证直到达标。**

把任务转化为可验证的目标：
- 「加校验」→「写出触发场景，烧录验证串口输出」
- 「修 bug」→「写一个能复现的故障步骤，修复后验证」
- 「重构 X」→「烧录验证改动前后行为一致」

多步骤任务，给出简洁计划：
```
1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]
3. [步骤] → 验证: [检查项]
```

验收条件**必须具体可观察**（串口日志、LED 行为、按键响应），不写"应该能工作"。

### 验收失败处理

1. 回滚改动
2. 记录复现场景（触发条件 + 实际现象 + 期望行为）
3. 修正后二次验证通过方可合入

### 排错原则

单元优先于集成。代码出错时，先在出错的模块（单个任务或单个驱动）内复现并修复，验证通过后再做整体联调。

### 边缘案例检查清单

- 硬件无响应（指纹不插 / SPI 掉线 / I2C NACK）
- 时序边界（Flash 跨页 / 队列满 / 定时器溢出）
- 并发冲突（双任务操作 Flash / 中断抢占）
- 输入异常（空指针 / 超长字符串 / 非法键值）
- **低电压供电异常（电池门锁核心场景）：** 电池欠压导致外设供电不稳、通信断续

---

**这些准则生效的标志：** diff 里无关改动变少、因过度设计导致的重写变少、澄清性问题出现在实现之前而不是出错之后。

## 5. 与 Skill 的映射

> 以上四条准则由以下 skill 在工程实践中强制执行。Agent 在项目中应主动 invoke 对应 skill。
> 具体任务→skill 映射见 `AGENTS.md` §17 Skill 路由表。

| 准则 | 强制执行 skill | 如何体现 |
|---|---|---|
| ① 先想清楚再写代码 | rainstorming、grill-me、writing-plans、domain-modeling | brainstorming 列出假设和方案 → grill-me 追问漏洞 → domain-modeling 对齐术语到 CONTEXT.md → writing-plans 产出可执行步骤。歧义判定标准触发前置确认流程 |
| ② 极简优先 | `codebase-design` | Deep module 方法论：小接口承载深实现——但保留嵌入式必要的超时返回、判空（嵌入式例外条款） |
| ③ 手术式改动 | `implement`、`request-refactor-plan` | implement 按 PRD/issue 驱动实现；request-refactor-plan 用于迁移/重构类大改动，拆分为安全增量提交 |
| ④ 目标驱动执行 | `tdd`、`verification-before-completion` | tdd 定义可烧录验证的成功标准 → verification 逐条跑 AGENTS.md §16 自检清单 → 验收失败则回滚 |
